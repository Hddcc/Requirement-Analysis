# 需求来源读取规范

## 基本原则

需求来源可以是一个文件、多个文件、目录或网页链接。先识别格式和可用工具，再选择读取方式。不能默认需求一定来自 Word，也不能只凭扩展名猜测内容结构。

内部记录每个来源的路径或地址、格式、更新时间、读取方式和有效范围。多个来源发生冲突时，结合用户最新说明判断优先级；无法判断且会影响后端设计时进入决策流程。

## 纯文本和标记文件

Markdown、TXT、RST、JSON、YAML、XML 等文本格式优先使用文本读取和结构化解析。保留标题层级、列表、代码块、表格、引用和字段层次。TXT 没有显式结构时，根据段落、编号和上下文提取需求，避免把示例文案误认为接口字段。

## 办公文档

DOC 和 DOCX 使用能够读取段落、标题、表格、批注、页眉页脚和嵌入图片的工具。XLS、XLSX 和 CSV 使用表格解析工具，保留工作表、表头、单元格类型、合并关系和备注。PPT 和 PPTX 需要读取页面文字、表格、备注及页面顺序。

## PDF 和图片

PDF 先提取文本，再根据版式复杂度渲染页面检查。包含流程图、原型、表格、批注或扫描页时，必须进行视觉检查或 OCR。PNG、JPG、WEBP 等图片使用视觉工具读取，记录图片与相邻文字或页面的关系。

## 网页和在线文档

网页链接使用当前 Agent 可用的浏览或抓取工具读取。保留页面标题、正文、表格和关联附件。遇到登录、权限或动态加载限制时，说明缺失范围并请求用户提供可访问文件，不能根据页面标题补写内容。

## 多来源合并

需求正文、原型、接口附件、补充说明和用户对话可以共同构成目标行为。先按来源拆解，再合并重复项并标记冲突。用户最新明确决策优先于旧附件，产品需求中的目标行为优先于历史技术方案，当前代码只用于说明现状和实现约束。
