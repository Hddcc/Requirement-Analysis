# 决策与交互规范

## 决策分类

发现问题后先判断它属于哪一类。

代码可发现事实应通过阅读仓库解决。产品需求已有明确答案的事项按需求执行。前端或客户端独立决策且不改变后端契约的事项进入联调清单。影响后端接口、数据、状态、兼容、成本或风险的事项进入阻塞决策。低风险且符合现有代码惯例的实现细节由 Agent 按最小改动原则决定，并在文档中说明依据。

## 候选方案表达

每个阻塞决策需要说明问题背景、代码证据、方案差异、改动范围、兼容和数据风险、测试成本、推荐方案及推荐理由。方案数量由实际情况决定，允许一个、两个或多个。用户可以选择、组合、否定或提供自由文本方案。

不要一次抛出大量低关联问题。优先处理会改变整体架构、协议和数据模型的决策，再处理局部实现。相互依赖的问题可以一起说明。

## 用户回复判断

结合最新消息和完整上下文进行语义判断，内部可归为以下状态：

- `SUPPLEMENT`：补充背景、约束或事实。
- `CORRECTION`：纠正 Agent 理解或前文信息。
- `QUESTION`：询问方案，仍未授权写入。
- `PARTIAL_DECISION`：解决部分问题。
- `READY_WITH_RECOMMENDATIONS`：允许按已展示的推荐项处理。
- `READY`：确认决策并允许生成。
- `REDIRECT`：改变需求、范围或目标文档。

分类依赖语义，不依赖固定关键词。用户在选择某个选项时附加条件，需要把附加条件一起写入决策台账。用户仍在连续补充时，继续接收信息，不能提前生成文档。

## 决策台账

内部为每个问题保存稳定标识、问题、证据、候选方案、推荐项、用户决定、受影响的需求和文档位置。需求变化后重新打开受影响的决策。最终文档只保留结论、理由和仍需要前端联调的事项，不呈现内部对话历史。
