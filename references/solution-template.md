# Solution 文档模板

Solution 负责解释需求为什么这样实现、当前代码具备什么、有哪些可选方案、最终选择及其风险。章节可以根据项目和需求调整，避免机械填充空章节。

建议结构如下：

# `<需求名称>`技术方案分析

写明需求来源和配套 implementation 文档路径。

## 1. 需求拆解

按用户场景解释产品目标、后端责任、前端或客户端责任和明确不在范围内的事项。建立需求到后端能力的对应关系。

## 2. 当前代码现状

写清真实入口、服务注册、分层、相似功能、可复用接口、配置、存储、消息和测试。重要结论带文件和函数证据。

## 3. 候选方案与最终选择

只保留真实存在过技术取舍的方案。说明每个方案的实现路径、优缺点、兼容和数据影响，再写最终选择及理由。已经由代码惯例自然确定的细节无需制造候选方案。

## 4. 总体方案和完整调用链

按功能域组织。每个功能写清请求来源、入口、业务层、配置和存储、转换、返回、消息或定时任务。MySQL、Redis、Lua 和消息队列应写进实际使用它们的功能链路。

## 5. 接口和协议设计

写请求、响应、字段语义、枚举、默认值、错误码、鉴权、兼容和调用方。协议字段需要符合当前项目风格。

## 6. 数据、配置和异步设计

说明表、索引、Redis key、TTL、Lua、消息结构、消费者、幂等、事务和历史数据。没有使用某项技术时不创建对应设计。

## 7. 前端和客户端联调

说明后端返回什么、调用方如何消费、状态和错误如何表现、哪些事项由客户端决定、哪些契约需要共同确认。

## 8. 风险、兼容和发布

说明旧接口、旧数据、缓存、消息、配置、灰度、回滚和观测风险。只写与需求和代码有关的风险。

## 9. 验收标准

把产品要求转换成可检查的接口、数据、兼容、测试和联调结果。
