# 项目分析规范

## 分析目标

在阅读产品需求前，建立足够完整的项目事实地图。分析结果用于约束方案，不能把本文件列出的技术当成项目已经使用的技术。

## 扫描顺序

先读取仓库根目录的 Agent 指令、README、构建文件、依赖清单、工作区配置和部署文件。随后定位程序入口、服务注册、路由、RPC、控制器、定时任务、消息消费者和插件入口。

继续定位项目实际使用的通信契约，例如 OpenAPI、Proto、IDL、HTTP 路由、事件结构、请求响应类型、错误码、鉴权和上下文读取方式。确认相关代码是否由其他仓库生成，以及当前仓库如何升级依赖和执行代码生成。

梳理业务层、领域层、页面聚合层、配置层、存储层、转换层、外部适配层和测试层。目录名称可能不同，应根据依赖方向和实际调用判断职责。

## 技术适用性清单

先建立当前项目的技术组件清单。每项组件记录证据位置、项目用途、与需求的关系和是否计划修改。清单可以包含通信框架、数据库、文件存储、缓存、搜索、脚本执行、消息或事件、任务调度、配置中心和外部平台，也可以不包含其中任何一类。

后续分析和审查只启用满足下列任一条件的检查项：当前代码真实使用；需求会触达；最终候选方案计划引入。项目未使用且方案未引入的技术不需要分析，也不能写入 solution 和 implementation。

## 必查内容

确认对象如何创建、依赖如何注入、服务如何挂载、接口如何转入业务层、业务返回和系统错误如何区分。查找至少一个与新需求最接近的完整功能，从入口跟到最终返回和异步后续处理。

项目存在持久化组件时，确认数据模型、事务或一致性边界、迁移方式、历史数据、索引或等价约束。项目存在缓存或键值存储时，确认键生成、数据结构、有效期、回源、幂等和原子操作。项目存在消息、事件或后台任务时，确认数据结构、生产和处理位置、挂载方式、延迟、重试、重复、乱序和异常记录。项目存在配置系统时，确认来源、结构、默认值、环境隔离、灰度、刷新和发布方式。

外部平台需要从当前项目的适配代码确认真实作用。相同平台在不同项目中可能承担支付、资格、内容、用户、消息或运营能力，字段和时序必须以当前调用代码为准。

测试方面需要确认测试框架、Mock 生成、测试数据、集成测试和常用验证命令。方案中的测试安排要沿用现有模式。

## 证据要求

内部记录每项重要事实的文件路径和符号名。方案中出现的现有函数、字段、表、key、消息和配置必须有证据。搜索不到时继续检查接口定义、实现、Mock 和测试；仍无法确认时将其列为未知事实，不能自行补全。

## 分析边界

系统分析应覆盖与需求可能相关的完整调用链。无需逐行解释 vendor、二进制、构建缓存和无关生成代码。发现工作区已有用户改动时保留这些改动，不回退、不覆盖，也不把它们误认为仓库基线。
