# Implementation 文档模板

Implementation 负责说明开发者按什么顺序、在哪些真实文件和函数中完成改动。所有标识必须来自代码或明确声明为新增。

建议结构如下：

# `<需求名称>`详细技术实现

写明需求来源和配套 solution 文档路径。

## 0. 改动清单

列出层次、文件、动作和目的。区分新增文件、修改文件、协议仓库、配置操作、数据操作和测试。

## 1. 协议和依赖

给出协议服务、RPC、请求响应、枚举、字段编号、校验、错误码、代码生成和依赖升级步骤。兼容规则需要写清。

## 2. 内部类型和配置

写出需要新增或修改的实体、配置结构、字段、标签、读取方法、默认值和发布方式。命名和类型沿用当前项目。

## 3. 接入层和服务挂载

说明入口文件、实现对象、函数签名、上下文和鉴权读取、日志、预发布标记、业务调用和返回方式。新增服务或任务时写清实际挂载位置。

## 4. 业务实现

按功能域写完整调用链。每一步说明调用者、被调用函数、参数、读取数据、业务判断、返回值和下一步。代码示例应贴合现有风格，避免用伪造 API 代替真实实现。

## 5. 数据和异步处理

根据技术组件清单逐项说明实际存在或计划引入的数据读写、缓存、原子操作、消息或事件、后台任务、一致性、幂等和失败处理。涉及历史数据时写清是否需要迁移或回填，以及上线顺序。某类组件不适用时省略对应内容。

## 6. 转换和返回

说明内部实体如何转换成协议结构，排序、过滤、默认值、空值和兼容字段如何处理，最终返回给哪个调用方。

## 7. 测试计划

按现有测试框架列出单测、Mock、存储测试、协议兼容测试和必要的集成验证。每项测试对应具体风险或验收要求。

## 8. 联调和发布清单

列出后端、前端、客户端、配置、数据和外部平台的联调内容，以及灰度、监控、告警、回滚和验收步骤。

实现步骤需要达到另一名熟悉该项目的开发者可以直接编码的精度。缺少决策时返回决策流程，禁止用占位符掩盖。
