# 产品需求技术文档 Skill

## 目标

根据当前代码仓库、产品需求原文、用户补充说明和用户最终决策，生成两份正式文档：

- `<需求标识>-solution.md`：解释需求、当前实现、候选方案、选型理由、总体设计、接口契约、数据设计、联调边界、风险和验收标准。
- `<需求标识>-implementation.md`：写清精确到文件、类型、函数、调用链、配置、存储、消息、测试和联调的实现步骤。

输出必须适配当前项目实际使用的框架、技术栈、目录分层、服务挂载、命名习惯、错误处理、依赖注入、配置体系、存储体系和测试方式。先分析代码，再拆解需求。不得把其他项目的做法直接套入当前仓库。

## 事实来源

区分现状事实和目标行为。

现状事实的可信顺序为：当前源代码、协议和数据结构；有效测试和部署配置；项目说明文档；代码注释和历史方案。目标行为的可信顺序为：用户最新明确决策；产品需求原文；用户早期补充；Agent 建议。

源代码与需求发生差异时，分别说明当前实现和目标行为。文档里的每个现有文件、函数、类型、字段、数据模型、数据标识、异步数据和配置都必须有仓库证据。计划新增的标识需要明确写成新增内容。

## 强制状态

执行期间维护以下内部状态：

- `project_facts`：代码确认的项目事实和证据位置。
- `technology_inventory`：当前项目真实存在、与需求相关或方案计划引入的技术组件。
- `requirement_items`：逐条拆出的产品要求。
- `open_decisions`：会影响后端设计且尚未解决的事项。
- `accepted_decisions`：用户已确认的方案和约束。
- `client_owned_questions`：前端或客户端独立决定且不阻塞后端的事项。
- `draft_findings`：对现有文档或临时方案的审查结论。
- `ready_to_write`：用户是否表达了可以生成或重构文档的意图。

每次收到用户回复后，结合完整上下文判断意图：

- 用户继续补充、纠正、质疑或追问时，更新上下文并继续分析。
- 用户回答部分问题时，更新决策台账并处理剩余事项。
- 用户修改需求或范围时，重新分析受影响的需求和代码。
- 用户表示“可以写”“没问题”“开始生成”时，检查阻塞事项是否清空。
- 用户表示“按推荐方案写”时，可以接受已经展示且未被反对的推荐项。
- 存在阻塞决策时，继续说明影响并等待用户决定。
- 阻塞决策清空且用户允许写入时，进入最终文档阶段。

交互没有固定轮数。不得仅靠关键词推进阶段，也不得因用户从预设选项中选了一个答案就忽略其自由文本补充。

## 工作流程

### 一、分析项目

完整阅读 `references/project-discovery.md`。先检查仓库级说明文件和 Agent 指令，再系统分析语言、框架、依赖、入口、服务注册、通信方式、分层、配置、数据组件、异步机制、外部平台、测试和相似功能。建立 `technology_inventory`，只记录有代码、配置或方案证据的组件。

能够从仓库确认的事实自行查证，不向用户提问。跳过 vendor、二进制、构建缓存和明显无关的生成产物。生成代码可以帮助定位协议和调用，业务结论应回到协议源、业务实现或测试。

### 二、拆解需求

完整阅读 `references/requirement-input.md`。需求来源可以是 Markdown、TXT、Word、PDF、表格、图片、网页链接或多个文件的组合。根据实际格式选择读取工具，保留标题、列表、表格、批注、图片和页面关系等会影响需求含义的信息。建立内部追踪关系：

`产品要求 -> 用户场景 -> 后端责任 -> 前端或客户端责任 -> 当前能力 -> 缺口 -> 候选方案 -> 验收方式`

主动识别需求没有直接说明的技术问题，包括协议归属、数据来源、幂等、并发、事务、历史数据、消息可靠性、配置默认值、鉴权、错误处理、兼容、发布和回滚。

### 三、讨论候选方案

完整阅读 `references/decision-workflow.md`。存在多个合理实现时，向用户说明每个方案的实现路径、代码适配程度、改动范围、数据风险、兼容影响、测试成本和推荐理由。允许用户选择、组合、修改或提出新方案。

前端或客户端独立决定的视觉、交互、分享渠道和渲染行为记入联调事项，不阻塞后端文档。会改变协议字段、枚举语义、接口时序、错误码、鉴权或数据含义的事项属于共同决策，需要用户确认。

### 四、形成临时文档并进行对抗审查

完整阅读 `references/solution-template.md`、`references/implementation-template.md` 和 `references/adversarial-review.md`。

目标文档已经存在时，读取现有内容并进行审查。目标文档尚不存在时，在分析上下文中形成临时文档。对抗审查完成前，不把临时版本写入最终路径。

审查前根据 `technology_inventory` 和最终候选方案建立适用检查项。只检查项目真实使用、需求会触达或方案计划引入的组件；不得因为参考文件列出了某类技术就假设项目使用它。

对抗审查发现的阻塞问题需要带代码证据、影响范围、候选处理方式和推荐意见交给用户。继续执行不限轮次的决策流程，直到所有后端阻塞问题解决。

### 五、写入正式文档

只有满足以下条件时才能创建或覆盖目标文档：

- 用户已表达可以生成或重构。
- 后端阻塞决策已经解决。
- 对抗审查已经完成。
- 临时文档已按最终决策修正。

直接创建或覆盖原有 solution 和 implementation 文件。禁止使用 v1、v1.1、新版、修订版等版本标识，也不在正文描述修改历史。最终文档呈现为一次完成的正式方案。

每项实现步骤都要写清：

`谁调用 -> 入口文件和函数 -> 下一层对象和函数 -> 读取或写入的数据 -> 返回值 -> 返回给谁 -> 是否触发异步处理`

### 六、最终验证

完整阅读 `references/writing-style-zh.md`，然后再次执行需求追踪和对抗审查。验证所有文件和现有标识能够从仓库找到，新增标识有明确声明，前后两份文档互相一致，没有未解决占位符和未经确认的结论。

只修改用户指定的两份文档。不得顺手修改业务代码、协议、配置、依赖或其他文件。

## 核心写作约束

- 中文为主，代码标识前补充中文含义。
- 使用直接、正向、自然的陈述。
- 避免“不是 xxx，而是 xxx”“不只是 xxx，而是 xxx”“不仅 xxx，而是 xxx”等先否定再肯定句式。
- implementation 必须精确到真实文件、类型、函数和调用关系。
- 优先复用现有能力，保持满足需求的最小改动。
- 不添加没有需求依据或真实风险依据的兜底、降级、抽象和扩展。
- 不把建议写成当前代码已经存在的事实。
- 不隐藏风险，不保留 `TBD`、`TODO`、`待定` 或未经确认的占位符。
- 明确前端和客户端的联调责任，纯客户端决策不要求后端用户代为决定。
