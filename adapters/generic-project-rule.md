# requirement-tech-docs 项目规则适配

当用户要求分析产品需求、生成技术方案、生成详细实现文档、审查 solution 或 implementation 时，必须执行以下规则：

1. 完整读取本项目中 `requirement-tech-docs/SKILL.md`。如果 Skill 位于其他目录，使用实际路径。
2. 按照 `SKILL.md` 的引用完整读取 `references` 下的项目分析、决策流程、文档模板、对抗审查和中文写作规范。
3. 先分析当前项目真实代码和需求原文，再讨论技术方案。
4. 用户仍在补充、纠正或追问时继续维护上下文，不提前生成最终文档。
5. 存在多个合理方案时说明实现路径、影响、风险和推荐理由，允许用户提供自由文本方案，交互轮数不设上限。
6. 正式写入前完成对抗式审查，把会影响旧代码、旧数据、协议、Redis、MySQL、Lua、消息和外部调用方的问题交给用户决定。
7. 后端阻塞决策解决且用户表达可以写入后，创建或覆盖 solution 和 implementation 文档。
8. 只修改目标文档，不顺手修改业务代码、协议、配置和其他文件。

如果当前 Agent 无法自动加载外部文件，应先读取这些文件并把规则保留在本次会话上下文中，再开始处理需求。
